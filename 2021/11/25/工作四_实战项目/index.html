Troy–web服务器实现

  • 简单介绍下项目？

    • 目的：这个项目主要的目的是对浏览器的链接请求进行解析处理，处理完之后给浏览器客户端返回一个响应报文和详细处理等。

    • 处理：服务器后端的处理方式使用socket通信，利用多路IO复用，可以同时处理多个请求，请求的解析使用预先准备好的线程池。

      使用模拟proactor模式，主线程充当异步线程，负责监听所有socket上的事件，监听到有事件之后，分配一个线程去处理该请求。

      睡眠在请求队列上的工作线程被唤醒进行处理，处理的方式采用状态机。

项目架构

Troy是一个基于Linux/C，专为转码集群通信的HTTP服务器，它具有以下特性：

  • 基于no-blocking & 多路IO复用模型，多线程更高效轻量
  • 支持Web前端部署

  • ^[1]^ sigaction(SIGPIPE, &sa, NULL)

    • BUG： 如果对方关闭了socket（fd1），进程还往fd1写数据，系统会给进程发生SIGPIPE信号。而SIGPIPE信号默认的signal handler 是终止进程，导致进程被终止。
    • 解决： 重新设置SIGPIPE信号 handle为 SIG_IGN （忽略信号的处理程序） ，这样就不会出现上述情况导致终止进程。

  • ^[2]^ open_listenfd(cf.port)

    • **主要用于初始化监听socket()**：socket → bind → listen
      1. 创建一个socket描述符 : socket()
      2. 把一个地址族中的特定地址赋给socket ：bind()
      3. 监听来连接socket：listen()

  • ^[3]^ make_socket_non_blocking(listenfd)

    在ET模式下（见①处设置），我们要将设置所有的socket为非阻塞模式（包括listendfd 和 clientfd），原因如下：

    • 为什么设置ET模式？

      • BUG：当线程处理某个clientfd时，数据未处理完此时依旧处于可读状态，LT模式下，触发epoll_wait回调事件会将该clientfd返回。按照图一右侧部分逻辑，该clientfd会交给新线程处理，这样就导致两个线程【同时】处理一个fd。

      • 解决：因此，我们设置所有fd为ET模式 ，只有有新数据来才返回这个clientfd。这样，数据在未处理完就不会返回这个clientfd。

    • 为什么设置ET + 非阻塞？

      • 阻塞模式： 阻塞模式下，如果无数据可读时read会返回0，会阻塞在read等待fd可读，不会执行后面逻辑。

      • 非阻塞模式：（1）read返回0，表示对端的写端关闭，准备结束进程；read返回**<0** ，（2）如果此时errno=EAGAIN ，表示暂无数据等待下次读取；（3）read返回**<0** 的其他情况，读取出错准备退出进程。

      • BUG（阻塞模式下）： ET模型下，由于数据可读也不会返回，保证读完数据要for死循环一次读完fd数据 （参见下图2，do_request中逻辑）。但是这样的话，当读完fd的数据时，由于是阻塞模式，read返回0阻塞在read，当前线程也不会结束。

        而现实情况，大多数是长连接，等待下一次连接时间不确定，线程便一直阻塞，得不到利用，而线程池中线程是有限的（n个），当第n+1个连接来时，便没有线程去处理。

      • 解决： 设置为 ET + 非阻塞 ，当read读取数据无论是否读取都立即返回。

        这样的话，read没读取到数据不会阻塞进程，而是返回该fd（等待下次epoll事件，新进程read继续上一次位置读取fd，因为fd上绑定了相应状态机记录了上次read位置），break循环准备退出处理函数，让进程可以执行其他连接。

  • ^[4]^ int epoll_wait(int epfd,struct epoll_event events,int maxevents,int timeout)

    • epfd：由epoll_create 生成的epoll专用的文件描述符；
    • epoll_event：用于回传代处理事件的数组；
    • maxevents：每次能处理的事件数；
    • timeout：等待I/O事件发生的超时值（ms）, -1永不超时，直到有事件产生才触发。 【timeout==某个正整数】表示等待这个正整数时间，没有事件发生也返回。这个timeout参数实际是find_timer()查询当前最近的定时器超时时间。

  • ^[5]^ While(1)

    ET模式下，读写事件只响应一次，用 while 循环抱住 accept 调用，处理完 TCP 就绪队列中的所有连接后再退出循环。否则的话一次只能处理一个连接，只有下次fd有新连接（ET模式条件满足），epoll_wait回调触发才能处理下一个链接。

    • 如何知道是否处理完就绪队列中的所有连接呢？accept 返回-1 && errno== EAGAIN 。

  • ^[6]^ int accept(int sockfd, struct sockaddr addr, socklen_t len）**

    • sockfd ：由socket函数返回的套接字描述符；
    • addr & addrlen：用来返回已连接的对端进程（客户端）的协议地址，如果我们对客户端的协议地址不感兴趣，可以把arrd和addrlen均置为空指针。

  • ^[7]^ EPOLLONESHOT

    • BUG : 即使设置了 ET模式 ，依旧有可能出现 两个线程【同时】处理一个fd 情况。

      • 当epoll_wait已经检测到socket描述符fd1，并通知线程A处理fd1的数据。如果【处理过程中】该fd1又有新的数据可读，会唤醒其他线程B处理fd1，那么就出现了两个工作线程同时处理fd1的情况 。

      处理完事件，重置这个epolloneshot事件，让这个fd可以被返回，从而被其它线程狐狸。

线程的同步机制有哪些？

现在流行的进程线程同步互斥的控制机制，其实是由最原始，最基本的4种方法实现的：

  • 临界区：通过多线程的互串行访问公共资源或一段代码，速度快，适合控制数据访问。
  • 互斥量（ 又称为互斥锁）：为协调共同对一个共享资源的单独访问而设计。只有拥有互斥对象的线程才有权限去访问系统的公共资源，因为互斥对象只有一个，所以能够保证资源不会同时被多个线程访问。
  • 信号量：为控制一个具有有限数量的用户资源而设计。它允许多个线程在同一时刻去访问同一个资源，但一般需要限制同一时刻访问此资源的最大线程数。
  • 事件：用来通知线程有一些事件已发生，从而启动后继任务的开始。

线程池中的工作线程是一直等待吗？

是的，等待新任务的唤醒。

你的线程池工作线程处理完一个任务后的状态是什么？

这里要分两种情况考虑

（1） 当处理完任务后如果请求队列为空时，则这个线程重新回到阻塞等待的状态

（2） 当处理完任务后如果请求队列不为空时，那么这个线程将处于与其他线程竞争资源的状态，谁获得锁谁就获得了处理事件的资格。

如果同时1000个客户端进行访问请求，线程数不多，怎么能及时响应处理每一个呢？

该项目是基于IO复用的并发模式。需要注意的是，不是一个客户连接就对应一个线程！如果真是如此，淘宝双12服务器早就崩了！

当客户连接有事件需要处理的时，epoll会进行事件提醒，而后讲对应的任务加入请求队列，等待工作线程竞争执行。如果速度还是慢，那就只能够增大线程池容量，或者考虑集群分布式的做法。

如果一个客户请求需要占用线程很久的时间，会不会影响接下来的客户请求呢，有什么好的策略呢?

会，因为线程的数量是固定的，如果一个客户请求长时间占用着线程资源，势必会影响到服务器对外的整体响应速度。解决的策略可以是给每一个线程处理任务设定一个时间阈值，当某一个客户请求时间过长，则将其置于任务请求最后，或断开连接。

多线程改为多进程怎么改？

使用for循环，每次都调用fork()创建6个子进程。

并发模型相关

简单说一下服务器使用的并发模型？

以Proactor模式为例的工作流程即是：

  1. 主线程充当异步线程，负责监听所有socket上的事件；

  2. 若有新请求到来，主线程接收之以得到新的连接socket，然后往epoll内核事件表中注册该socket上的读写事件 ；

  3. 如果连接socket上有读写事件发生，主线程从socket上接收数据（Troy是工作线程），并将数据封装成请求对象插入到请求队列中 ；

  4. 所有工作线程睡眠在请求队列上，当有任务到来时，通过竞争（如互斥锁）获得任务的接管权。

reactor、proactor、主从reactor模型的区别？

（Troy）Reactor模式：要求主线程（I/O处理单元）只负责监听文件描述符上是否有事件发生（可读、可写），若有，则立即通知工作线程，将socket可读可写事件放入请求队列，读写数据、接受新连接及处理客户请求均在工作线程中完成。(需要区别读和写事件)

Proactor模式：主线程和内核负责处理读写数据、接受新连接等I/O操作，工作线程仅负责业务逻辑（给予相应的返回url），如处理客户请求。

主从Reactor模式：核心思想是，主反应堆线程只负责分发Acceptor连接建立，已连接套接字上的I/O事件交给sub-reactor负责分发。其中 sub-reactor的数量，可以根据CPU的核数来灵活设置。下图即是其工作流程：

主反应堆线程一直在感知连接建立的事件，如果有连接成功建立，主反应堆线程通过accept方法获取已连接套接字，接下来会按照一定的算法选取一个从反应堆线程，并把已连接套接字加入到选择好的从反应堆线程中。主反应堆线程唯一的工作，就是调用accept获取已连接套接字，以及将已连接套接字加入到从反应堆线程中。

你用了epoll，说一下为什么用epoll，还有其他复用方式吗？区别是什么？

  • 对于select和poll来说，所有文件描述符都是在用户态被加入其文件描述符集合的，每次调用都需要将整个集合拷贝到内核态；epoll则将整个文件描述符集合维护在内核态，每次添加文件描述符的时候都需要执行一个系统调用。系统调用的开销是很大的，而且在有很多短期活跃连接的情况下，epoll可能会慢于select和poll由于这些大量的系统调用开销。
  • select使用线性表描述文件描述符集合，文件描述符有上限；poll使用链表来描述；epoll底层通过红黑树来描述，并且维护一个ready list，将事件表中已经就绪的事件添加到这里，在使用epoll_wait调用时，仅观察这个list中有没有数据即可。
  • select和poll的最大开销来自内核判断是否有文件描述符就绪这一过程：每次执行select或poll调用时，它们会采用遍历的方式，遍历整个文件描述符集合去判断各个文件描述符是否有活动；epoll则不需要去以这种方式检查，当有活动产生时，会自动触发epoll回调函数通知epoll文件描述符，然后内核将这些就绪的文件描述符放到之前提到的ready list中等待epoll_wait调用后被处理。
  • select和poll都只能工作在相对低效的LT模式下，而epoll同时支持LT和ET模式。
  • 综上，当监测的fd数量较小，且各个fd都很活跃的情况下，建议使用select和poll；当监听的fd数量较多，且单位时间仅部分fd活跃的情况下，使用epoll会明显提升性能。

【易忘】什么是ET（边缘触发）、LT（水平触发）？ET、LT优缺点？

ET模式：状态变化才触发

  • 缺点：应用层业务逻辑复杂，容易遗漏事件，很难用好。

  • 优点：相对LT模式效率比较高。一触发立即处理事件。

LT模式：可写可读就会触发

  • 优点：编程更符合用户直觉，业务层逻辑更简单。

  • 缺点：效率比ET低。

应用：

  • LT适用于并发量小的情况，ET适用于并发量大的情况。

    ET在通知用户之后，就会将fd从就绪链表中删除，而LT不会，它会一直保留，这就会导致随着fd增多，就绪链表越大，每次都要从头开始遍历找到对应的fd，所以并发量越大效率越低。ET因为会删除所以效率比较高。

HTTP报文解析相关

用了状态机啊，为什么要用状态机？

  • 过程：客户端发出链接，到达服务器，服务器这端先用函数一次性把所有请求读到缓冲区buffer。状态机主要用于解析客户端请求，状态机调用解析请求行，请求头，请求内容三部分函数进行解析（每次从buffer中读取n个字符，然后进入for循环从buffer解析一个字符，维持一个状态机）。解析结束后函数生成响应报文，该函数会根据不同的网址url产生不同响应体。最后通过write函数里套接字的传输方式把响应体传给客户端。

  • 为什么能使用：由于在HTTP报文中，每一行的数据由\r\n作为结束字符，空行则是仅仅是字符\r\n。因此，可以通过查找\r\n将报文拆解成单独的行进行解析，项目中便是利用了这一点。

  • 为什么使用：状态机主要用于对客户端请求的处理，其中有三种状态，处理请求行，处理请求头，处理请求体，有了状态机，一方面可以避免多种状态同时发生造成混乱

    1. 更可靠：另一方面保证了状态处理结束后能够正确的进行状态转移，相比只有if进行判断更加安全可靠；

    2. ET非阻塞模式：

      • 为了让一次读完fd数据，设置fd为ET模式，for 循环read直到返回-1才返回。而且循环read必须设置文件描述符为非阻塞，否则最后一次read缓存区无数据返回0，会阻塞（等待fd可读），这个时候read不能处理其它描述符。

      • 由于设置了非阻塞，数据可能分包到达 ，read=-1返回后，实际并没有读完数据。因此要设置一个状态机，读到需要的请求数据再结束。参考Nginx精简实现，每一次读取一个字符，记录此时读取的位置（状态）。

【易忘】定时器相关

为什么要用定时器？具体过程？

处理定时任务，或者非活跃连接，节省系统资源。

说一下定时器的工作原理？

服务器就为各事件分配一个定时器。该项目使用SIGALRM信号来实现定时器，首先每一个定时事件都处于一个升序链表上，通过alarm(10)函数周期性10s触发SIGALRM信号，而后信号回调函数利用管道通知主循环，主循环接收到信号之后对最小堆的定时器进行处理：若一定时间内无数据交换则关闭连接。

注意，本项目信号处理函数仅仅发送信号通知程序主循环，将信号对应的处理逻辑放在程序主循环中，由主循环执行信号对应的逻辑代码。

更具体实现原理：

定时器处理过程：

  • 初始定时器key = 当前时间+过期时间
  • 触发处理事件，如果key < 当前时间，删除定时器。
最小堆优化？说一下时间复杂度和工作原理？

  • 最小堆以每个定时器的过期时间进行排序，最小的定时器位于堆顶，当SIGALRM信号触发回调函数时执行过期定时器清除，如果堆顶的定时器时间过期，则删除，并重新建堆，再判定是否过期，如此循环直到未过期为止。

webbench是什么？介绍一下原理

是一款轻量级的网址压力测试工具，可以实现高达3万的并发测试。

  • 原理： 父进程fork若干个子进程，每个子进程在用户要求时间或默认的时间内对目标web循环发出实际访问请求，父子进程通过管道进行通信，子进程通过管道写端向父进程传递在若干次请求访问完毕后记录到的总信息，父进程通过管道读端读取子进程发来的相关信息，子进程在时间到后结束，父进程在所有子进程退出后统计并给用户显示最后的测试结果，然后退出。
